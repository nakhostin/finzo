# وب‌اپ آفلاین مدیریت مالی شخصی (Offline Personal Finance PWA)

## Context

کاربر در حال حاضر تمام درآمدها، بدهی‌ها، اقساط، بستانکاری‌ها و مخارج روزمره را در یک فایل اکسل (`docs/Loans.xlsx`) ثبت می‌کند. بررسی فایل نشان داد ساختار آن یک شیت سالانه با ستون‌های ماه (فروردین..اسفند) برای هر بدهی/درآمد، یک شیت template برای سال بعد، یک دفتر بدهی جداگانه برای یک شخص خاص، و یک لیست خرید ساده است. این ساختار "پهن" (هر ردیف = یک آیتم، هر ستون = یک ماه) امکان چک‌لیست کردن، پیگیری وضعیت هر پرداخت به‌صورت مجزا، گزارش‌گیری چندساله، و مدیریت چک/دارایی ارزی را نمی‌دهد.

هدف: بازنویسی کامل به یک وب‌اپ حرفه‌ای، کاملاً آفلاین (PWA نصب‌شدنی)، با تقویم جلالی و رابط RTL فارسی، که علاوه بر معادل‌سازی قابلیت‌های اکسل، موارد جدید زیر را هم پوشش دهد: چک‌های صادره/دریافتی، دفتر بستانکاری/بدهی به تفکیک اشخاص، دسته‌بندی مخارج (خانه، همسر، کادو، خودرو، سفر، پس‌انداز)، و دارایی‌های غیرریالی (دلار، لیر، طلا، نقره) با **تاریخچه کامل تراکنش‌های خرید/فروش** برای محاسبه سود/زیان.

تصمیمات از قبل با کاربر هماهنگ شده (تغییر ندهید):
- **پشته فنی**: React + Vite + TypeScript، به‌صورت PWA آفلاین‌محور و نصب‌شدنی (service worker + manifest).
- **بک‌آپ**: فقط Export/Import با JSON (نه اکسل).
- **دارایی ارزی/طلا/نقره**: مدل تاریخچه کامل تراکنش (buy/sell lots)، نه فقط موجودی لحظه‌ای.
- ذخیره‌سازی کاملاً محلی در مرورگر (IndexedDB)، بدون سرور.

## تصمیم کلیدی معماری داده: الگوی Template + Occurrence

به‌جای کپی ساختار "ستون به ازای هر ماه" اکسل، هر آیتم تکرارشونده (بدهی/قسط/درآمد) به دو مفهوم جدا می‌شود:
- **`RecurringItem`**: الگو/قانون تکرار (عنوان، روز سررسید، اولویت، مبلغ، بازه فعال بودن) — به‌ندرت تغییر می‌کند.
- **`LedgerEntry`**: وقوع مشخص برای یک سال+ماه جلالی خاص (مبلغ برنامه‌ریزی‌شده/واقعی، وضعیت پرداخت) — همان واحد چک‌لیست‌شدنی روزمره.

تراکنش‌های یک‌باره (کادو، خرید خاص) صرفاً `LedgerEntry` بدون `recurringItemId` هستند. این مدل هم چک کردن مستقل هر ماه را ممکن می‌کند، هم گزارش‌گیری چندساله را به یک کوئری ساده تبدیل می‌کند، هم غلتاندن به سال جدید را بدون ورود دستی مجدد انجام می‌دهد.

## مدل داده (`src/types/entities.ts`)

موجودیت‌های اصلی (فیلدهای کلیدی):
- `Category` — عنوان/عنوان فارسی، نوع (income/expense/both)، parentId برای زیردسته (مخارج خانه > برق/آب)، رنگ/آیکون.
- `Person` — طرف حساب، تعمیم‌یافته از شیت "بدهی های سینا" به هر تعداد شخص.
- `Account` — حساب بانکی/نقد، موجودی اولیه؛ موجودی جاری = مجموع تراکنش‌های مرتبط.
- `RecurringItem` — عنوان، نوع (`debt|installment|income|expense|receivable`)، categoryId، personId، accountId، priority، dueDay (۱ تا ۳۱، در زمان تولید clamp می‌شود)، defaultAmount یا installmentSchedule (برای وام‌های نامساوی)، بازه startYear/Month تا endYear/Month (باز = ongoing).
- `LedgerEntry` — لینک به RecurringItem (یا one-off)، jalaliYear/Month/dueDay، amountPlanned/Actual، status (`pending|paid|partial|skipped|overdue`)، paidJalaliDate.
- `Cheque` — شماره چک، accountId، direction (issued/received)، counterpartyPersonId، amount، تاریخ صدور/سررسید جلالی، status (`in-hand|deposited|cleared|bounced|cancelled`) + تاریخچه وضعیت.
- `AssetType` — کد (USD/TRY/GOLD/SILVER)، واحد (دلار/لیر/گرم).
- `AssetLot` — تراکنش خرید/فروش دارایی: assetTypeId، direction، quantity، unitPriceIRR، تاریخ — پایه محاسبه میانگین بهای تمام‌شده و سود/زیان.
- `RateSnapshot` — نرخ لحظه‌ای دستی‌واردشده برای هر دارایی، با تاریخ (چون API آنلاین در دسترس نیست).
- `ShoppingItem` — عنوان، مقدار، قیمت تخمینی/واقعی، status (`wishlist|planned|purchased|cancelled`)، priority — معادل شیت Buy.
- `JalaliYear` — رکورد کمکی برای پیگیری اینکه occurrence های کدام سال/ماه تولید شده‌اند (idempotency).

تصمیمات ساده‌سازی آگاهانه (باید مستند بماند، عوض نشود): مبالغ به‌صورت عدد صحیح **ریال** ذخیره می‌شوند (مطابق واحد فایل اکسل اصلی)؛ محاسبه بهای تمام‌شده دارایی‌ها با روش **میانگین موزون** (نه FIFO)؛ حسابداری تک‌طرفه (نه دو-طرفه کامل).

## مدیریت چندساله

`RecurringItem` مستقل از سال است (بازه باز/بسته). تابع `generateOccurrencesForYear(year)` برای هر آیتم فعال، `LedgerEntry` معادل هر ماه را با کلید یکتای `(recurringItemId, year, month)` upsert می‌کند (idempotent). trigger ها: (۱) بارگذاری خودکار برنامه در صورت عقب‌بودن سال جاری، (۲) دکمه صریح "شروع سال جدید" در تنظیمات، (۳) اصلاح یک آیتم که فقط ماه‌های آینده را دوباره تولید می‌کند و به گذشته/پرداخت‌شده‌ها دست نمی‌زند. سال‌های قدیمی چون `LedgerEntry` دائمی است همیشه قابل گزارش‌گیری می‌مانند.

## گزارش‌ها (`/reports`)

۱) خلاصه داشبورد (بدهی/درآمد/مانده/پس‌انداز ماه جاری) ۲) ارزش خالص در طول زمان (حساب‌ها + دارایی ارزی به نرخ روز − بدهی + بستانکاری) ۳) چک‌لیست اولویت پرداخت‌ها (قابل درگ) ۴) جریان نقدی به تفکیک دسته (خانه/همسر/کادو/خودرو/سفر) ۵) تقویم سررسیدهای پیش‌رو (اقساط + چک) ۶) دفتر هر شخص با مانده حساب ۷) عملکرد دارایی (میانگین بها، سود/زیان محقق‌نشده/محقق‌شده) ۸) وضعیت چک‌ها (در جریان/پاس/برگشتی) ۹) مقایسه سال‌به‌سال ۱۰) لیست خرید در برابر پس‌انداز موجود.

## فازبندی پیاده‌سازی

- **فاز ۰ — Scaffold**: Vite+React+TS، Tailwind RTL، Dexie schema اولیه، فونت وزیرمتن، `domain/jalali.ts`. خروجی: اپ آفلاین با پوسته RTL فارسی بالا می‌آید.
- **فاز ۱ — هسته دفترکل**: CRUD کامل RecurringItem، موتور `recurrence.ts`، چک‌لیست ماه جاری (تیک پرداخت‌شده)، داشبورد با جمع ماه + ۷ روز آینده. معادل شیت Year-1405.
- **فاز ۲ — اشخاص و چک**: CRUD شخص + دفتر شخصی، CRUD چک + گردش وضعیت + لینک به LedgerEntry.
- **فاز ۳ — دارایی‌ها (دلار/لیر/طلا/نقره)**: AssetLot خرید/فروش، RateSnapshot، محاسبه سود/زیان.
- **فاز ۴ — لیست خرید**: معادل شیت Buy با وضعیت و اولویت.
- **فاز ۵ — گزارش‌ها**: ۱۰ نمای بالا با Recharts.
- **فاز ۶ — UX چندساله**: دکمه "شروع سال جدید"، تاپ‌آپ خودکار، postpone آیتم.
- **فاز ۷ — PWA و بک‌آپ نهایی**: service worker، نصب، `navigator.storage.persist()`، export/import JSON نسخه‌بندی‌شده با zod، تست کامل در حالت آفلاین/پرواز.

## ریسک‌ها

- سال کبیسه جلالی و طول متغیر اسفند (۲۹/۳۰ روز) — هرگز hardcode نشود، همیشه از `jalaali-js` استفاده شود؛ روز سررسید باید در زمان تولید clamp شود.
- ثبات واحد پول (ریال) در تمام موجودیت‌ها برای جلوگیری از خطای اعشاری در اعداد بزرگ.
- قفل‌کردن روش میانگین موزون برای بهای تمام‌شده دارایی — تغییر بعدی آن گزارش‌های تاریخی را خراب می‌کند.
- مهاجرت schema در Dexie فقط رو-به-جلو (افزودن فیلد اختیاری، نه تغییر/حذف).
- idempotency تولید occurrence و "فقط ماه‌های آینده re-generate شوند، نه گذشته".
- کش service worker باید همه فایل‌های استاتیک (از جمله فونت) را precache کند وگرنه آفلاین واقعی نیست.
- ریسک پاک شدن IndexedDB توسط مرورگر تحت فشار حافظه → `navigator.storage.persist()` + یادآوری دوره‌ای بک‌آپ JSON چون سروری برای بازیابی وجود ندارد.

- خاموش کردن شبکه (DevTools → Offline) بعد از یک بار build+preview و تأیید بارگذاری کامل اپ.
- Export گرفتن JSON، پاک کردن IndexedDB از DevTools، Import مجدد و تطبیق داده‌ها.
- بررسی محاسبه سود/زیان دارایی با چند تراکنش خرید/فروش نمونه دلار.
